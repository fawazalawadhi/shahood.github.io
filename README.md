# shahooda — منصة تصميم دعوات المناسبات

مشروع Flutter Web لإنشاء دعوات إلكترونية (زفاف، خطوبة، مولود، تخرج، وأي مناسبة)
عبر قوالب جاهزة قابلة للتخصيص، مع معاينة حية أثناء التصميم، وحفظ التصميم كقالب،
ونشره كرابط.

## كيف تشغّل المشروع عندك

```bash
flutter pub get
flutter run -d chrome
```

> ملاحظة: هذا المشروع أُنشئ خارج بيئة تحتوي Flutter SDK، لذلك لم يتم تشغيله
> أو اختباره فعلياً بعد. راجع القسم أدناه (خطوات لازمة قبل أول تشغيل).

## خطوات لازمة قبل أول تشغيل

1. **خط Tajawal**: نزّل ملفات الخط من
   https://fonts.google.com/specimen/Tajawal وضع:
   - `Tajawal-Regular.ttf`
   - `Tajawal-Medium.ttf`
   - `Tajawal-Bold.ttf`

   داخل `assets/fonts/` (الأسماء يجب أن تطابق تماماً ما في `pubspec.yaml`).

2. **صور القوالب الافتراضية (Placeholders)**: ضع أي 3 صور مؤقتة بأسماء:
   - `assets/images/placeholder_wedding.jpg`
   - `assets/images/placeholder_newborn.jpg`
   - `assets/images/placeholder_engagement.jpg`

   (هذه تُستخدم في `TemplateProvider.seedIfEmpty()` كقوالب تجريبية أولية).

3. `flutter pub get` لتنزيل الحزم.

    template_gallery_screen.dart    معرض القوالب مع فلترة حسب المناسبة

## نظام صفحة التفاصيل (بعد الكشف) - أسلوب numinds

بعد ما يضغط الضيف على صورة الغلاف ويشتغل الفيديو، تظهر **صفحة تفاصيل تسكرول
لتحت** فيها الأقسام التالية بالترتيب اللي يحدده المصمم:

1. **شعر / آية / عبارة ترحيبية** - نص حر
2. **أسماء أصحاب المناسبة** - نص تفصيلي حر
3. **التاريخ والوقت** - منتقي تاريخ ووقت، يُعرض بصيغة عربية
4. **الموقع** - اسم القاعة + رابط خرائط جوجل اختياري
5. **عداد تنازلي** - يُحسب تلقائياً من التاريخ المحدد، يتحدث كل ثانية
6. **تأكيد الحضور (RSVP)** - أزرار "سأحضر/اعتذر" + عدد المرافقين (اختياري تفعيله)

**كل قسم:**
- له **مفتاح تشغيل/إيقاف (On/Off)** في لوحة التحكم يمين شاشة التصميم
- **قابل لإعادة الترتيب بالسحب** (`ReorderableListView`) - الترتيب يُحفظ فوراً
- محتواه محفوظ حتى لو مطفّى، فيقدر المستخدم يجهزه ثم يفعّله لاحقاً

عند الضغط "حفظ كقالب"، تظهر نافذة فيها اسم القالب + خيار **"قالب عام (أدمن)"**:
- لو غير مفعّل → يُحفظ كقالب خاص بهذا المستخدم فقط
- لو مفعّل → يظهر للجميع في معرض القوالب (هذا ما يُستخدم من شاشة الأدمن)

## آلية المعاينة الحية

`DesignProvider` هو `ChangeNotifier` يحمل كل بيانات الدعوة الحالية (نص، ألوان،
صورة، فيديو...). أي تغيير من أي حقل بالواجهة يستدعي دالة في `DesignProvider`
ثم `notifyListeners()`، فيعيد `Consumer<DesignProvider>` داخل `LivePreview`
بناء نفسه فوراً بالقيم الجديدة — بدون أي إعادة تحميل يدوية.

## الحالة الحالية مقابل الخطوات القادمة

**تم إنجازه في هذا التسليم (بيانات محلية في الذاكرة):**
- هيكلة كاملة للمشروع + الثيم بالألوان المطلوبة
- شاشة تصميم بمعاينة حية عبر Provider
- معرض قوالب مفلتر حسب نوع المناسبة
- لوحة أدمن تستخدم نفس شاشة التصميم + حفظ كـ"قالب عام" يظهر للجميع
- رفع صورة غلاف وفيديو خاص بكل دعوة من قبل المستخدم
- **صفحة تفاصيل كاملة بعد الكشف** (أسلوب numinds): شعر/آية، أسماء، تاريخ ووقت،
  موقع، عداد تنازلي حي، تأكيد حضور RSVP
- **تحكم كامل بكل قسم**: تشغيل/إيقاف + إعادة ترتيب بالسحب
- حفظ التصميم الكامل (كل الأقسام ومحتواها وترتيبها) كقالب باسم يختاره المستخدم
- توليد رابط نشر مبدئي (شكل: `shahooda.com/i/xxxx`)

**الخطوة القادمة المقترحة (عند تأكيدك):**
- ربط **Firebase** فعلياً:
  - `Firestore` بدل القوائم المحلية في `TemplateProvider` (حفظ دائم للقوالب والدعوات)
  - `Firebase Storage` لرفع الصور/الفيديوهات (بدل مسار الملف المحلي)
  - `Firebase Hosting` لتفعيل روابط `shahooda.com/i/xxxx` فعلياً على الدومين
  - `Firebase Auth` بسيط لتمييز حسابات المستخدمين (لكل واحد قوالبه المحفوظة)
- شاشة تسجيل دخول/حساب للمستخدم العادي (لحفظ قوالبه الخاصة)
- حماية لوحة الأدمن بكلمة مرور/صلاحية خاصة

أخبرني إذا تبي أبدأ بربط Firebase الآن، أو أطوّر أي شاشة من الشاشات الحالية أكثر
قبل ما ننتقل للخطوة التالية.
